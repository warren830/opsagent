# Design: OpsAgent 多租户隔离 + 阿里云支持

Generated by /office-hours on 2026-03-20
Branch: main
Repo: opsagent
Status: APPROVED
Mode: Startup (Intrapreneurship)

## Problem Statement

OpsAgent 是一个通过 IM（Teams/Slack/飞书）查询云基础设施的 AI 机器人。目标用户是客户团队的开发人员，每个团队拥有独立的 clusters、AWS 账号和阿里云账号。当前架构是单租户的——所有资源平铺在一起，没有团队间的数据隔离。同时，阿里云支持仅在配置中以注释形式存在，尚未实现。

已有人准备使用该产品，当前阻塞上线的两个核心问题是：多租户隔离和阿里云支持。

## Demand Evidence

- 多个团队的开发人员提出过类似需求（共同反馈，非单个具体人）
- 需求已经到了"有人要用"的阶段
- 黄灯信号：用户当前没有 workaround（连烂脚本都没写），说明痛点可能还需要通过上线后的实际使用来验证强度

## Status Quo

用户目前没有系统化的解决方案。查询基础设施状态可能靠手动登录 console、问同事、或者凭经验。没有统一的跨账号、跨云查询工具。

## Target User & Narrowest Wedge

- **目标用户：** 客户团队的开发人员
- **核心场景：** 在 IM 里用自然语言查询自己团队的 clusters、AWS/阿里云资源状态
- **隔离要求：** 团队 A 不能看到团队 B 的资源
- **最小 wedge：** 当前已有功能 + 多租户隔离 + 阿里云基础查询能力

## Constraints

- 现有架构基于 ECS Fargate + Claude Code CLI + Bedrock
- 阿里云需要 aliyun CLI 或 SDK 集成
- 每个团队通过 IM 频道/群组区分身份
- 运维成本不能随团队数线性增长（排除了每团队独立部署方案）

## Premises

1. 目标用户是客户团队的开发人员，需要查询自己团队的资源但不能看到别的团队的
2. 现有架构是单租户的，没有团队隔离，这是当前最大的架构缺口
3. 阿里云支持是必要的，不能只做 AWS
4. 已经有人要用了，重点是多租户隔离 + 阿里云支持，而不是继续加新功能

## Approaches Considered

### Approach A: 配置驱动的软隔离
在现有架构上加一层 tenant 配置（`config/tenants.yaml`），定义每个团队能访问哪些 AWS 账号、阿里云账号、clusters。根据 IM 消息来源匹配 tenant，查询时只返回该 tenant 范围内的资源。

- Effort: M (human: ~1周 / CC: ~30分钟)
- Risk: 中——隔离靠配置正确性，配错可能泄漏
- Pros: 最快上线，改动最小，现有功能都能复用
- Cons: 隔离不彻底，阿里云 CLI 集成需要额外工作
- Reuses: 现有所有代码，仅添加过滤层
- Completeness: 6/10

### Approach B: Tenant-aware 全链路改造（推荐）
从消息接收到 Claude 查询全链路注入 tenant context。每个查询进入 Claude 时，system prompt 动态注入该 tenant 的资源范围。阿里云通过 aliyun CLI 集成。Admin UI 加 tenant 管理页面。

- Effort: L (human: ~2周 / CC: ~1小时)
- Risk: 低——隔离在架构层面实现
- Pros: 完整的多租户 + 多云支持，一步到位
- Cons: 改动面较大，需要测试多租户场景
- Reuses: 现有 adapter 层、Admin UI 框架、知识库机制、cluster discovery 模式
- Completeness: 9/10

### Approach C: 每个团队独立部署
每个客户团队部署一个独立的 OpsAgent 实例（独立 ECS Service），通过 CDK context 参数化部署。

- Effort: M (human: ~1周 / CC: ~30分钟)
- Risk: 低——物理隔离
- Pros: 最强隔离，现有代码几乎不用改
- Cons: 运维成本高（N 个团队 = N 个 ECS Service），更新需要滚动 N 个实例
- Reuses: 现有所有代码，CDK stack 参数化
- Completeness: 7/10

## Recommended Approach

**方案 B: Tenant-aware 全链路改造**

理由：AI 工具让全链路改造的边际成本极低（~1小时 CC 时间），而方案 A 的隔离不彻底在多团队场景下是定时炸弹，方案 C 的运维负担会随团队数线性增长不可接受。

### 具体实现要点

**1. Tenant 数据模型 (`config/tenants.yaml`)**
```yaml
tenants:
  team-alpha:
    name: "Alpha 团队"
    channels:  # IM 频道/群组 → tenant 映射
      teams: ["channel-id-1"]
      feishu: ["chat-id-1"]
      slack: ["channel-id-1"]
    aws:
      accounts: ["111111111111", "222222222222"]
      clusters: ["alpha-prod-eks", "alpha-staging-eks"]
    alicloud:
      accounts:
        - name: "alpha-production"
          region: "cn-hangzhou"
          access_key_env: "ALPHA_ALICLOUD_AK"
          secret_key_env: "ALPHA_ALICLOUD_SK"
      clusters: ["alpha-ack-prod"]
  team-beta:
    name: "Beta 团队"
    channels:
      teams: ["channel-id-2"]
    aws:
      accounts: ["333333333333"]
    alicloud:
      accounts:
        - name: "beta-production"
          region: "cn-shanghai"
          access_key_env: "BETA_ALICLOUD_AK"
          secret_key_env: "BETA_ALICLOUD_SK"
```

**2. 全链路 tenant context 注入**
- `message-handler.ts`: 根据消息来源的 channel/chat ID 解析 tenant
- `claude-client.ts`: system prompt 动态注入 tenant 的资源范围
- 知识库: 添加阿里云查询指引（aliyun CLI 用法）
- `Dockerfile`: 安装 aliyun CLI

**3. Admin UI tenant 管理**
- 新增 Tenants 标签页，CRUD 管理租户配置
- 显示每个租户的资源范围、关联频道

**4. 安全边界**
- Claude system prompt 明确限制：只允许查询该 tenant 配置中的账号和 clusters
- foreach-account.sh 增加账号白名单参数
- 审计日志记录 tenant context

## Open Questions

1. **阿里云认证方式：** 每个团队用独立的 AK/SK？还是通过 RAM 角色跨账号？需要确认客户的阿里云账号结构
2. **IM 频道映射粒度：** 一个频道对应一个团队？还是可能一个用户在多个团队？
3. **阿里云覆盖范围：** 第一版需要支持哪些阿里云服务？（ECS、ACK、SLB、RDS...）
4. **混合查询：** 用户会不会问"我的团队在 AWS 和阿里云上一共有多少个 cluster"这种跨云问题？

## Success Criteria

1. 团队 A 在 IM 中查询时，只能看到自己的 AWS 账号和阿里云资源
2. 团队 B 的资源对团队 A 完全不可见
3. 阿里云基础查询可用（至少：ECS 实例列表、ACK 集群状态）
4. Admin UI 可以管理 tenant 配置
5. 第一个团队上线并实际使用

## Dependencies

- 阿里云 AK/SK 或 RAM 配置（需要客户提供）
- 各团队的 IM 频道/群组 ID
- 确认阿里云账号结构（单账号多 RAM 还是多独立账号）

## The Assignment

**在写任何代码之前，做这一件事：** 找到那个"一类人"里最积极的一个人（最好能说出名字），和他坐下来 15 分钟，问他三个问题：
1. 你现在每天/每周查基础设施状态的时候，具体在查什么？（不是"查集群状态"，而是具体到"查 alpha-prod 的 EKS 版本是不是 1.29"）
2. 你团队用的是 AWS 还是阿里云还是都用？具体有几个账号？
3. 如果我下周给你一个在 Teams/飞书里直接查的工具，你最想先查什么？

这 15 分钟会帮你确认 Open Questions 里的大部分问题，也会让你的第一个 demo 精准命中真实需求。

## What I noticed about how you think

- 你说"有人主动提过需求"但给不出具体名字——这说明需求是真的，但你还没有和具体用户建立深度连接。这是内部创业最危险的盲区：你在为一个模糊的群体做产品，而不是为一个具体的人解决一个具体的问题。
- 你说"就是现在已经做的这些"是最小 demo——这说明你对产品已有的能力有信心，但也意味着你可能在没有用户反馈的情况下就铺了很多功能（multi-provider、scheduled jobs、Admin UI、知识库体系等）。先让一个团队用起来，他们的反馈会告诉你下一步该做什么。
- 你明确知道需要多租户和阿里云——这是好的产品判断，说明你理解用户的真实约束（每个团队有自己的账号和集群，不能互相看到）。
- 你选了完整方案 B 而不是走捷径——这说明你在意做对的事而不是做快的事，这在内部创业中很重要，因为第一个版本的架构决策会跟你很久。
